Design notes

dim_periodicity — promoted from degenerate

Previously periodicity_code was a degenerate dimension on every fact. Promoting it to dim_periodicity adds periodicity_name (e.g. "Day-to-date"), period_type, accumulation_days, and sort_order — useful for display labels, sorting in reports, and filtering by period class without hardcoding codes in SQL.

dim_attribution_breakdown vs dim_breakdown_node

dim_attribution_breakdown is the type dimension (sector, core, global) — 3 rows, stable, no hierarchy. dim_breakdown_node is the hierarchy member dimension — one row per node within a breakdown (investment_grade → emerging_markets → asia → india). The parent_node_sk self-reference builds the tree; use bridge_breakdown_path at query time for efficient rollups at any level.

FK ↑ — cross-fact foreign keys

fact_attribution_breakdown_level.attribution_account_sk → fact_attribution_account links each breakdown row to its parent fund row, enabling drill-through without re-joining all dimensions. Similarly fact_attribution_security.attribution_breakdown_sk → fact_attribution_breakdown_level chains the three grains into a navigable hierarchy.

dim_security scoped to security fact only

dim_security is a FK only on fact_attribution_security. Account-level and breakdown-level facts don't reference it — individual securities are not a meaningful concept at those coarser grains. dim_account, dim_date, dim_periodicity, dim_attribution_breakdown, and dim_breakdown_node are conformed across all three facts.
